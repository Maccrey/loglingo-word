# WordFlow Tasklist

## 1. 문서 해석 기준

- 현재 저장소에는 구현 코드가 없으므로, 이 문서는 `PRD.md`, `README.md`, `UI.md`를 기준으로 한 초기 구축용 작업 분해 문서다.
- 우선순위는 문서에 명시된 런칭 플랜을 따른다.
- 1차 범위: MVP
- 2차 범위: MLP
- 3차 범위: 운영/품질/확장
- 모든 기능 태스크는 "작게 구현하고 바로 테스트"를 원칙으로 한다.
- 모든 구현 완료 후 커밋 메시지는 한국어로 작성한다.

## 2. 구현 원칙

- 한 태스크는 가능한 한 하나의 책임만 가진다.
- 한 커밋은 하나의 사용자 가치 또는 하나의 기술 단위만 담는다.
- UI 구현 전에 도메인 모델과 테스트 가능한 로직을 먼저 만든다.
- 비즈니스 로직은 UI와 분리한다.
- AI 기능은 비용 통제를 위해 fallback 경로를 항상 둔다.
- i18n 문자열은 하드코딩하지 않고 locale JSON으로 관리한다.

## 3. 권장 초기 구조

- `apps/web`: Next.js UI
- `services/core`: 학습, 메모리 엔진, 퀴즈, 점수 규칙
- `services/ai`: AI 대화, 교정, 추천
- `services/payment`: Polar 결제 및 권한 처리
- `services/leaderboard`: 주간 랭킹 집계
- `packages/shared`: 타입, 상수, 스키마, 공통 유틸
- `tests/unit`: 순수 함수/도메인 단위 테스트
- `tests/integration`: Firebase, API, Webhook, 화면 흐름 통합 테스트

## 4. 테스트 기본 원칙

- 목표 커버리지는 `README.md` 기준으로 80% 이상을 유지한다.
- 최소 단위 테스트는 "순수 함수", "상태 전이", "권한 분기", "실패 처리", "fallback 처리"를 우선한다.
- 화면 테스트는 모든 UI를 전부 스냅샷하지 말고 핵심 상호작용만 검증한다.
- 외부 연동은 실제 호출 대신 mock/stub로 검증하고, 별도 통합 테스트에서 계약을 확인한다.

## 5. 완료 정의

- 태스크 산출물이 코드에 반영되어야 한다.
- 명시된 최소 단위 테스트가 모두 통과해야 한다.
- 관련 i18n 문자열이 추가되어야 한다.
- 실패 케이스가 최소 1개 이상 테스트되어야 한다.
- `Tasklist.md`에서 해당 태스크를 완료 체크한 뒤 한국어 커밋 메시지로 커밋한다.

## 9. 커밋 규칙

- 커밋 메시지는 반드시 한국어로 작성한다.
- 한 커밋에는 한 태스크만 담는 것을 원칙으로 한다.
- 커밋 전에 반드시 `Tasklist.md`에서 완료한 태스크의 상태를 `[x] 완료`로 갱신한다.
- 아직 끝나지 않은 태스크는 `[ ] 진행 중` 또는 미완료 상태로 유지한다.
- 형식 예시:
- `온보딩 상태 전이 로직 구현`
- `플래시카드 학습 화면 구현`
- `리더보드 집계 서비스 구현`
- `Polar 결제 웹훅 처리 구현`
